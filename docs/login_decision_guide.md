**문서 버전**: 1.0

**작성자**: UX 컨설턴트

## 1. 문서 목적

이 문서는 TouchAd 서비스에 **로그인 기능을 언제, 왜, 어떻게 도입해야 하는지**에 대한  
UX 관점의 판단 기준을 제공합니다.

---

## 2. 현재 상태: 왜 로그인을 도입하지 않았는가?

### 2.1 Phase 1의 핵심 목표

```
목표: 세일즈 리드 생성 (Lead Generation)
수단: 가치 체험 → 관심 신호 포착
```

### 2.2 로그인 없이 달성 가능한 것

| 목표 | 로그인 없이 가능? | 현재 구현 |
|------|------------------|-----------|
| 서비스 가치 전달 | ✅ 가능 | Landing Page |
| 맞춤형 데이터 제시 | ✅ 가능 | 주소 기반 시뮬레이션 |
| 관심 신호 포착 | ✅ 가능 | Inquiry 제출 |
| 연락처 수집 | ✅ 가능 | Inquiry 폼 |
| 세션 추적 | ✅ 가능 | sessionStorage + DB |

### 2.3 로그인이 없을 때의 장점

1. **마찰 최소화**: 진입 장벽 없음
2. **빠른 가치 증명**: "먼저 보여주고, 나중에 요청"
3. **이탈률 감소**: 회원가입 포기로 인한 이탈 방지
4. **개발 효율**: 인증 시스템 구축 비용 절감

## 3. 로그인 도입이 필요해지는 UX 조건

### 3.1 트리거 정의

로그인 도입이 **필요해지는 명확한 조건**:

| # | 트리거 조건 | 신호 | 판단 기준 |
|---|------------|------|-----------|
| 1 | **반복 방문 패턴** | 동일 세션/IP에서 여러 시뮬레이션 실행 | 30% 이상이 2회+ 방문 |
| 2 | **데이터 저장 요구** | "이 결과 저장하고 싶어요" 피드백 | 직접 요청 발생 |
| 3 | **자동화 병목** | Admin 수동 처리 한계 도달 | 일 20건 이상 Inquiry |
| 4 | **리마케팅 필요** | 비활성 리드 재접촉 필요 | 전환율 하락 |
| 5 | **프리미엄 티어** | 유료 기능 구분 필요 | 비즈니스 모델 확장 |

### 3.2 각 트리거 상세 설명

#### 트리거 1: 반복 방문 패턴

**현상:**
```
User A: 1월 15일 시뮬레이션 실행
User A: 1월 17일 다시 방문 → 기존 결과 없음 → 재실행
```

**문제점:**
- 동일 User를 반복 리드로 잘못 카운트
- User 경험 단절 (이전 결과 찾을 수 없음)

**해결책:** 로그인으로 세션 연속성 확보

---

#### 트리거 2: 데이터 저장 요구

**현상:**
```
User: "이 분석 결과 좋은데, 저장해두고 나중에 다시 보고 싶어요"
User: "동료한테 공유하고 싶은데요"
```

**구현 검토:**
- 시뮬레이션 결과 영구 저장
- 마이페이지에서 히스토리 확인
- 공유 링크 생성

---

#### 트리거 3: 자동화 병목

**현상:**
```
Admin: "하루에 Inquiry 50개씩 들어오는데, 일일이 상태 변경하기 힘들어요"
```

**필요 기능:**
- Self-service 상태 확인 (User가 직접)
- 자동 응답 시스템
- 예약 시스템

**전제조건:** User 식별 = 로그인

---

#### 트리거 4: 리마케팅 필요

**현상:**
```
Inquiry 제출 후 비활성 리드 30일 경과
재접촉하고 싶으나, 연락처 외 접점 없음
```

**구현 검토:**
- 이메일/푸시 알림
- 새 분석 결과 자동 발송

---

#### 트리거 5: 프리미엄 티어

**현상:**
```
무료: 시뮬레이션 1회/월
유료: 무제한 + 경쟁사 분석 + 트렌드 리포트
```

**구현 검토:**
- 결제 연동
- 사용량 추적
- 권한 관리

---

### 4.2 권장 도입 지점

> **Phase 2 로그인 도입 시점:**  
> **"결과 저장" 기능 출시 직전**

**이유:**
1. User에게 명확한 **가치 교환** 제공 (로그인하면 저장 가능)
2. 기존 가치 체험 흐름 유지 (로그인 없이도 시뮬레이션 가능)
3. 점진적 전환 (새 기능부터 로그인 요구)

## 5. 로그인 도입 시 UX 권장사항

### 5.1 가치 교환 원칙

```
❌ 나쁜 UX: "로그인하세요" → 그래야 뭐라도 볼 수 있음
✅ 좋은 UX: "로그인하면 결과를 저장하고 다시 볼 수 있어요"
```

### 5.2 Progressive Disclosure 패턴

```
Phase 2 Recommended Flow:
─────────────────────────────────────────────────────
[Landing] → [시뮬레이션] → [Dashboard] → [결과 저장?] → [로그인]
                              │                           │
                         여기까지는                    여기서만
                         로그인 불필요                 로그인 요청
```

### 5.3 소셜 로그인 우선

현재 코드에 "카카오 로그인" 버튼이 준비되어 있습니다.

**권장:**
- **1순위**: 카카오 로그인 (한국 User 편의성)
- **2순위**: 네이버 로그인
- **3순위**: 이메일 로그인 (폴백)

## 6. 구현 로드맵 (참고용)

```
Phase 2 로그인 도입 로드맵
─────────────────────────────────────────────────────

Stage 1: 데이터 수집 (현재 → +2주)
 ├── 재방문 패턴 분석 (session 데이터)
 ├── User 피드백 수집
 └── Admin 병목 지표 확인

Stage 2: 의사결정 (+2주 → +4주)
 ├── 트리거 조건 충족 여부 판단
 └── 도입 결정 또는 지연

Stage 3: 구현 (결정 후)
 ├── 카카오 OAuth 연동
 ├── User 테이블 연결
 ├── "결과 저장" 기능 개발
 └── 마이페이지 구현
```
